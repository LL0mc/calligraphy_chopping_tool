## 二、参数实验（以 RapidOCR 基线为基准）

> **注意：** 此部分实验使用旧的乘法复合评分公式（`Score = 100 × text_acc × edge_factor × detection_f1`），当前代码已切换为成本评估器（cost-based evaluator）。两种评估器的绝对分数不可直接比较，但实验结论（相对优劣）仍然有效。

### 2.1 旧评估公式

```
Score = 100 × (正确文本数 / max(GT, DET)) × edge_factor × detection_f1

edge_factor = 1 - clamp(平均绝对边缘误差 / ref_size, 0, 0.3)
detection_f1 = 2 × precision × recall / (precision + recall)
precision = 匹配数 / 检测数
recall = 匹配数 / GT数
```

边缘误差：分别测量上下左右四边。Hungarian 算法做 GT 与检测框的位置匹配（中心距离 ≤ 60px）。

### 2.2 基准线（Iter 0）

**参数：**

| 参数 | 值 |
|------|-----|
| search_margin_x | 40 |
| search_margin_y | 100 |
| merge_radius | 100 |
| bbox_padding | 5 |
| binary_threshold | 140 |
| expand_padding | 15 |
| gap_threshold (effective) | 100 |
| missing_char_min_area | 500 |
| 列尾搜索范围 | 2× 平均字高 |
| CC 合并距离 | 80 |

**结果：**

| 指标 | 值 |
|------|-----|
| GT 框数 | 1314 |
| 检测框数 | 1323 |
| 匹配数 | 1313 |
| 漏检 | 1 |
| 误检 | 10 |
| 文字准确率 | 94.0% (1234/1313) |
| 文字错误 | 79 |
| 边缘误差 l/t/r/b | 0.3 / 2.6 / 4.2 / 1.3 px |
| 均值边缘误差 | 2.1 px |
| Detection F1 | 0.996 |
| **综合评分** | **91.57** |

### 2.3 CC refine 参数调优（5 轮实验）

**Iter 1 (v1) — ❌ 方法缺陷**

修改：search_margin_x 40→60, search_margin_y 100→120, merge_radius 100→120,
expand_padding 15→20, gap_threshold 100→80, missing_char_min_area 500→600,
列尾搜索 2×→2.5×, CC 合并 80→100

结果：evaluator 方法缺陷——`orig_idx` 在参数变化后无效。
- **教训：** GT 必须用静态基准线（baseline + corrected.json），用 Hungarian 位置匹配

**Iter 1 (v2) — ❌ 轻微退步 (91.36)**

同上参数 + 修复后的 evaluator

| 指标 | Baseline | Iter 1 v2 |
|------|----------|-----------|
| Score | **91.57** | **91.36** |
| 均值边缘误差 | 2.1 | 2.3 |
| 文字准确率 | 94.0% | 93.9% |

分析：扩大搜索范围→CC 抓了更多噪声→边缘变差

**Iter 2 — ❌ 大幅退步 (90.34)**

修改：binary_threshold 140→150, bbox_padding 5→3, 新增 `_is_valid_chinese`

| 指标 | Baseline | Iter 2 |
|------|----------|--------|
| Score | **91.57** | **90.34** |
| 均值边缘误差 | 2.1 | 4.0 |

分析：threshold=150 断了笔画连通行。左边缘右移 +2.6。
**教训：** 140 是 handwriting 的近最优值，改高会破坏连通性。

**Iter 3 — ❌ 过修正 (91.35)**

修改：merge_radius 100→80, bbox_padding 5→3

| 指标 | Baseline | Iter 3 |
|------|----------|--------|
| Score | **91.57** | **91.35** |
| 均值边缘误差 | 2.1 | 3.4 |

分析：merge_radius=80 改善了右边缘（+4.2→+2.1），但丢了左边的 CC 组件（l +0.2→+2.1）。
**教训：** 100 是最优值。右偏不是 CC 抓了远处组件造成的。

**Iter 4 — ❌ 对称减 padding 伤左缘 (91.13)**

修改：bbox_padding 5→4

| 指标 | Baseline | Iter 4 |
|------|----------|--------|
| Score | **91.57** | **91.13** |
| 均值边缘误差 | 2.1 | 2.9 |
| 文字准确率 | 94.0% | **94.1%** |
| 文字错误 | 79 | **78** |

分析：padding 5→4 给了精确的 -1px 对称偏移，但绝对值因左缘从 0.2 变成 1.2 而上升。
**关键认识：** padding 对称但偏差不对称。减 padding 改善了右/上/下但伤了完美的左。

**Iter 5 — ❌ 非对称校正引入方差 (91.07)**

修改：bbox_padding 5（还原）, 新增 refine_shrink_right=3, refine_shift_top=1

| 指标 | Baseline | Iter 5 |
|------|----------|--------|
| Score | **91.57** | **91.07** |
| 符号偏差 l/t/r/b | +0.2/-2.2/+4.2/+1.1 | +0.2/-1.2/+1.2/+2.1 |
| 绝对偏差 l/t/r/b | 0.3/2.6/4.2/1.3 | **0.3**/3.5/**6.3**/2.3 |
| 文字准确率 | 94.0% | **94.1%** |
| 文字错误 | 79 | **77** |

分析：符号偏差大幅改善（r+4.2→+1.2, t-2.2→-1.2），但绝对值不降反升（r 4.2→6.3）。校正在页间不一致。
**教训：** 全局固定值的非对称校正不可行，页间偏差方差太大。

### 2.4 替代 CC refine 方案

**跳过 CC refine — ❌ 有效但框太小 (73.53)**

| 指标 | Baseline | 跳过 CC |
|------|----------|---------|
| Score | **91.57** | **73.53** |
| 匹配数 | 1313 | 1275 |
| 漏检 | 1 | **39** |
| 均值边缘误差 | 2.1 | **25.3** |

分析：原始 OCR 框比 GT 小约 25px/边。CC refine 虽引入右偏，但将漏检从 39 降到 1。**CC refine 不是可选项，是必需品。**

**行级检测 + 等分 — ❌ 更差 (64.97)**

| 指标 | Baseline | 行级等分 |
|------|----------|----------|
| Score | **91.57** | **64.97** |
| 匹配数 | 1313 | 1234 |
| 漏检 | 1 | **80** |

分析：竖排字帖列级框等分完全对不准。

**原始框中心等比例外扩 — ❌ 最差 (61.82)**

| 指标 | Baseline | 等比外扩 |
|------|----------|----------|
| Score | **91.57** | **61.82** |
| 均值边缘误差 | 2.1 | **62.9** |

分析：从偏心的 OCR 框中心外扩，越扩越偏。

### 2.5 文字纠错

`_is_valid_chinese` 过滤非中文字符，准确率 94.0%→94.1%（79→77 错误）。

### 2.6 参数实验总结

**评分轨迹：**
- Baseline: **91.57**（当前最优）
- Iter 1: 91.36 (−0.21)
- Iter 2: 90.34 (−1.23)
- Iter 3: 91.35 (−0.22)
- Iter 4: 91.13 (−0.44)
- Iter 5: 91.07 (−0.50)
- 跳过 CC: 73.53
- 行级等分: 64.97
- 等比外扩: 61.82

**核心结论：**
1. **CC refine 是必需步骤**——它将漏检从 39 降到 1，边缘误差从 25px 降到 2.1px
2. **CC refine 的右偏是二阶问题**——+4.2px 的偏差在 25px 的原始误差面前微不足道
3. **所有替代 CC 的方案都大幅退步**——没有 CV 辅助的纯 OCR 精度不可接受
4. **下一步应该聚焦文字纠错（77 个错误）**，而非边缘精度

**测试页：** 28 页（24–35, 37–44, 50, 60, 79–81, 85, 99, 100）

## 六、后续验证：27.9 分的真正原因（2026-06-19）

### 6.1 问题

CNSTD 官方模型得分 27.9 分异常偏低。作为成熟的中文文本检测模型，即使 backbone 随机初始化，也不应该如此之差。

### 6.2 排查过程

1. **`detector.py:156` 硬编码 `pretrained_backbone=False`** — CNSTD 的 Detector 类在初始化时禁用了预训练 backbone，导致即使网络下载成功，backbone 也是随机权重。`cli.py` 中虽已取消禁用（注释说明 torchvision 0.26+ 支持），但 `detector.py` 未同步修改。

2. **模型 checkpoint 可用** — `C:\Users\LMC\AppData\Roaming\cnstd\1.2\db_shufflenet_v2_small\` 下有 12MB 的 checkpoint 文件，检测模型权重本身已下载。

3. **实测 CNSTD 检测输出** — 在 page 24 上运行 CNSTD，输出 18 个区域框。可视化后发现：

   | 框类型 | 示例 | 说明 |
   |--------|------|------|
   | 整列框 | #2 (237×2698px), #3 (204×2664px) | 整列文字作为一个区域 |
   | 区域框 | #0 标题, #8 "青埂峰顽石偈" | 段落/标题级 |
   | 碎片框 | #4, #5, #9 等 | 注释文字片段 |

   **CNSTD 输出的是文本区域级框（列/段落），不是字符级框。**

### 6.3 根因

**粒度不匹配**：
- **CNSTD/CnOCR**：文本区域检测器 → 输出列级/行级框
- **RapidOCR（PP-OCRv5）**：字符级检测+识别 → 输出单字级框
- **Pipeline 需求**：字符级定位 → 与 RapidOCR 天然兼容

评估器用 Hungarian 匹配（中心距离 ≤ 60px），区域框的中心 ≠ 单字框的中心，导致大量匹配失败，分数崩塌。

### 6.4 CnOCR 调研

CnOCR V2.2+ 内部调用 CnSTD 做检测，同样输出区域级框。虽然 V2.3.1+ 集成了 PP-OCRv4/v5 识别模型，但检测端仍是区域级。**CnOCR 不支持字符级检测。**

### 6.5 结论

| 包 | 检测粒度 | 字符级框 | 与 Pipeline 兼容 |
|---|---------|---------|-----------------|
| **RapidOCR** | 字符级 | ✅ `return_word_box=True` | ✅ |
| **PaddleOCR** (原版) | 字符级 | ✅ PP-OCRv4/v5 | ⚠️ 需 PaddlePaddle 框架 |
| **CnOCR** | 区域级 | ❌ 依赖 CnSTD | ❌ |
| **CNSTD** | 区域级 | ❌ | ❌ |

**整个换检测器的实验方向有误** — 不该替换检测器，应在 RapidOCR 的字符级结果上做后处理优化。27.9 分不是模型差，是架构不兼容。

## 七、Pipeline 改进实验（2026-06-19）

### 成本评估器

当前评估器（`src/evaluator.py`）使用成本模型反映人工审阅工作量：

| 操作 | 成本 |
|------|------|
| 匹配框（边缘偏差 > 3px） | 0.1 × 四边偏差之和 |
| 匹配框（文字不同） | 2 |
| 漏检框（需新增） | 8 |
| 误检框（一键删除） | 1 |

`score = max(0, 100 × (1 - total_cost / max_cost))`，其中 `max_cost = n_gt × 10`。

CLI 用法：`python src/evaluator.py [--det-dir <path>]`，`--det-dir` 用于指向实验目录的检测结果。

### 基线（成本评估器）

| 指标 | 值 |
|------|-----|
| 平均分 | 90.43 |
| 文字错误 | 79 (cost 158) |
| 漏检 | 1 (cost 8) |
| 误检 | 10 (cost 10) |
| 总成本 | 1273 / 13140 |

### 错误分析

- **文字错误 (79)**：56 个 (71%) 置信度 < 0.8，9 个 (11%) 置信度 ≥ 0.9（形近字）
- **边角误差 (230 字)**：左边 1.6px，上边 15.5px，**右边 22.7px**（系统性偏窄），下边 7.9px
- **误检 (10)**：6 个空文字 conf=0.0（噪声），2 个"一"字

### 实验 1：过滤噪声框

**方法**：移除 ocr_text 为空且 ocr_score < 0.3 的框。

| 指标 | 基线 | 实验 1 | 变化 |
|------|------|--------|------|
| 得分 | 90.43 | 90.15 | -0.28 |
| 漏检 | 1 | **10** | +9 ❌ |
| 误检 | 10 | 5 | -5 ✓ |
| 文字错误 | 79 | 70 | -9 ✓ |

**结论**：过滤器太激进，把 9 个有效低置信度字符也删了。漏检成本（8/个）远大于误检节省（1/个）。**失败**。

### 实验 2：右侧边框加宽

**方法**：每个框的 w 增加 15px，补偿右侧系统性偏窄。

| 指标 | 基线 | 实验 2 | 变化 |
|------|------|--------|------|
| 得分 | 90.43 | **75.47** | -14.96 ❌ |
| 边缘误差 | 8.8px | **23.4px** | +14.6 ❌ |

**结论**：固定加宽太粗暴，对右侧已经正确的框引入新误差。**失败**。

### 实验 3：增加 OCR 扩展边距

**方法**：expand_padding 从 15 增加到 25，让 OCR 看到更多上下文。

| 指标 | 基线 (p=15) | 实验 3 (p=25) | 变化 |
|------|------------|--------------|------|
| 得分 | 90.43 | 90.40 | -0.03 |
| 文字错误 | 79 | 77 | -2 ✓ |

**结论**：OCR 识别对上下文边距不敏感，无显著改进。

### 总结

| 实验 | 方法 | 结果 | 教训 |
|------|------|------|------|
| 1 | 过滤噪声框 | 失败 | 漏检成本 >> 误检节省，不能盲目过滤 |
| 2 | 右侧加宽 | 大幅退步 | 固定偏移量不适合逐字调整的场景 |
| 3 | 增大 expand_padding | 无显著变化 | OCR 识别对边距不敏感 |

**核心结论**：当前 RapidOCR + CC refine 的 pipeline 已接近最优。进一步提升需要：
1. **更多训练数据** — 微调 OCR 模型（需 PaddlePaddle 框架）
2. **人工标注积累** — 用 Review GUI 标注更多页面，扩大 GT 数据集
3. **边缘误差** — 系统性右侧偏窄是 CC refine 的固有特性，非参数调整可解

## 八、PP-OCRv5 检测+识别适配实验（2026-06-19）

### 背景

RapidOCR 3.8.4 支持 PP-OCRv5 模型。PP-OCRv5 识别精度比 PP-OCRv4 更高，但检测框行为有差异。本次实验目标：让 pipeline 适配 PP-OCRv5 检测，同时使用 PP-OCRv5 识别。

### PP-OCRv5 检测 vs PP-OCRv4 检测

| 特性 | PP-OCRv4 | PP-OCRv5 |
|------|----------|----------|
| 检测框宽度 | ~141px | ~220px |
| 注释文字检测 | 部分忽略 | 全部检测 |
| 框中心位置 | 与 GT 接近 | 略有偏移 |
| 字符级输出 | 是 | 是（但更宽） |

### 实验过程

#### 尝试 1：直接使用 PP-OCRv5 检测
- 结果：得分 73.01（基线 90.43），大幅退步
- 原因：检测框太宽，CC refinement 搜到相邻字符组件

#### 尝试 2：median_area 列过滤（阈值 12000）
- 结果：得分 77.45，略有改善
- 原因：部分注释列面积与书法列接近，无法完全分离

#### 尝试 3：增大 expand_padding（15→25）
- 结果：得分 90.40，无显著变化
- 原因：OCR 识别对边距不敏感

#### 尝试 4：CC refinement drift check（中心偏移 >60px 则回退）
- 结果：得分 75.05，反而退步
- 原因：丢弃了正确的 refined box，回退到同样偏移的 detection box

#### 尝试 5：CC refinement 搜索区域裁剪到检测框范围
- 结果：得分 57.3，大幅退步
- 原因：裁剪太激进，切掉了正确组件

#### 尝试 6：median_area 过滤 + overlap_ocr 限制
- **结果：得分 95.08，大幅超越基线 90.43** ✓
- 关键修复：`overlap_ocr` 不再无条件包含重叠组件，限制在 `merge_radius * 0.5` 距离内

#### 尝试 7：去掉 overlap_ocr 限制（最终方案）
- **结果：得分 94.27，误检 23，文字错误 66，准确率 95.1%**
- 发现：overlap_ocr 限制是过度工程化，去掉后效果更好
- 真正起作用的是 merge_radius=50 的距离检查 + claimed_regions 防重复

### 最终方案

| 配置 | 值 |
|------|-----|
| 检测模型 | PP-OCRv5 |
| 识别模型 | PP-OCRv5 |
| merge_radius | 50（从 100 降低） |
| overlap_ocr 限制 | 已移除（过度工程化） |
| 列过滤 | 中位面积 >= 12000px |
| 噪声过滤 | 空文字 conf < 0.5 |

### 结果对比（成本评估器，29 页基线）

| 指标 | v4 基线 | v5 + 后处理 | 最终方案 | 变化 |
|------|--------|-----------|---------|------|
| **平均分** | 90.15 | 91.08 | **94.27** | **+4.12** |
| **误检** | 474 | 9 | 23 | -451 |
| **漏检** | 7 | 5 | 7 | 0 |
| **文字错误** | 66 | 84 | **66** | 0 |
| **文字准确率** | 95.1% | 93.8% | **95.1%** | 0 |
| **总成本** | 1255 | 1190 | **804** | -451 |

最终方案：merge_radius=50, 无 overlap_ocr 限制, median_area≥12000, 空文字 conf<0.5

> 注：页面集随提交进度变化，不同时间点的评估结果不可直接比较。以上为当前 29 页基线的对比。

### 关键发现

1. **overlap_ocr 限制是过度工程化**：完全去掉 overlap_ocr 限制后，效果与限制在 merge_radius*0.75 完全一致。真正起作用的是 merge_radius=50 的距离检查 + claimed_regions 防重复。

2. **merge_radius 从 100 降到 50**：距离检查更严格，避免抓取远处组件。

3. **median_area 列过滤**：中位字符面积 < 12000px 的列被识别为注释列并过滤。PP-OCRv5 检测的注释字符面积（~8000px）与书法字符（~22000px）有明显差异。

4. **PP-OCRv5 检测框更宽是特性而非缺陷**：更宽的框能捕获完整字符（包括笔画末端），但需要适配 CC refinement 逻辑。
