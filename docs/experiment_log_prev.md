# CNSTD/CNOCR 替代 RapidOCR 实验总结

> 日期: 2026-06-18
> 分支: `feat/cnocr-finetune`
> 最终结论：方案不可行，RapidOCR 保持最优。

## 实验目标

用 CNSTD（检测）+ CNOCR（识别）替代 RapidOCR，提升书法字帖 OCR 精度。

## 旧评估方法及问题

### 评估公式

```
Score = 100 × (正确文本数 / max(GT, DET)) × edge_factor × detection_f1
edge_factor = 1 - clamp(平均绝对边缘误差 / ref_size, 0, 0.3)
detection_f1 = 2 × precision × recall / (precision + recall)
```

### 问题

1. **乘法复合**：三项相乘导致分数过度偏低（每项 0.95 → 总分 85.7）
2. **漏检和误检等价**：F1 对称计算，但实际漏检成本（需新增框）远高于误检（一键删除）
3. **无法反映审阅工作量**：用户花时间调框和改字，但评估器不区分这两种成本

## 实验过程与发现

### 1. CNSTD 检测 + RapidOCR 识别

**结果**：得分 27.9（基线 91.1），大幅退步。

**根因排查**：
1. `detector.py:156` 硬编码 `pretrained_backbone=False`，backbone 随机初始化
2. 即使修复 backbone，CNSTD 输出的是**区域级框**（列/段落），不是字符级框
3. Pipeline 需要字符级定位，与 CNSTD 架构不兼容

**关键发现**：CNSTD 不支持字符级检测。它输出文本区域（整列文字作为一个框），而 pipeline 的列过滤、CC refinement 都基于字符级框设计。

### 2. RapidOCR 检测 + CNOCR 识别

**结果**：准确率 59.5%（基线 93.7%）。

**典型错误**：
- 形近字：峰→蜂、无→元、枉→神、谁→淮
- 非中文：一→T、石→a、可→m、红→2

**结论**：CNOCR 官方模型在印刷体上训练，对书法字帖泛化能力有限。

### 3. CNSTD 微调（22 页数据）

**数据**：22 页 ~800 个字符框（ICDAR 多边形格式），80/20 分训练/验证。

**PL 2.6.1 兼容性修复（12 处）**：dataset.py、trainer.py、metrics.py 等多处 API 变更。

**训练结果**：

| 方案 | Page 24 得分 |
|------|-------------|
| 无预训练骨干 | 27.9 |
| 有预训练骨干 | 68.3 |
| RapidOCR 基线 | **91.1** |

预训练骨干带来 +40.4 分提升，但 22 页数据仍不够。

### 4. CNOCR 微调（22 页数据）

**结果**：准确率从 59.5% 降到 42.7%（灾难性遗忘）。

22 页书法数据让模型忘记印刷体知识，微调后反而更差。

### 5. CnOCR 不支持字符级检测

CnOCR V2.2+ 内部调用 CnSTD 做检测，输出区域级框。虽然 V2.3.1+ 集成了 PP-OCRv4/v5 识别模型，但检测端仍是区域级。**CnOCR 不支持字符级检测。**

### 6. CNSTD 27.9 分的真正原因

**不是模型差，是架构不兼容**：
- CNSTD 输出区域级框（列/段落），Pipeline 需要字符级框
- 评估器用 Hungarian 匹配（中心距离 ≤ 60px），区域框中心 ≠ 单字框中心
- 匹配失败导致分数崩塌

## 最终结论

| 包 | 检测粒度 | 字符级框 | 与 Pipeline 兼容 |
|---|---------|---------|-----------------|
| **RapidOCR** | 字符级 | ✅ | ✅ |
| **PaddleOCR** (原版) | 字符级 | ✅ | ⚠️ 需 PaddlePaddle |
| **CnOCR** | 区域级 | ❌ | ❌ |
| **CNSTD** | 区域级 | ❌ | ❌ |

**整个换检测器的实验方向有误** — 不该替换检测器，应在 RapidOCR 的字符级结果上做后处理优化。

## 教训

1. **架构兼容性优先于模型性能**：CNSTD/CnOCR 虽然模型成熟，但输出粒度与 pipeline 不兼容
2. **数据量是微调的瓶颈**：22 页不够任何检测/识别模型的微调
3. **灾难性遗忘**：小数据集微调大模型会导致原有能力退化
4. **评估方法需要匹配业务场景**：旧评估器的乘法公式不适合衡量人工审阅成本
